## 一、项目定位

### 1.1 平台定位

**无敌 (Invencia)** 是一个叙事驱动的文字 RPG 平台，承载多个 AI 智能体驱动的"世界"。每个世界拥有独立的世界观、规则体系与叙事风格，由专属 AI 智能体全程驱动。玩家通过自由文本输入做出抉择，每段叙事以开放式钩子结尾。

### 1.2 世界架构

```
无敌 (Invencia) 平台
  ├── 修仙世界（天道智能体）← MVP 首个世界
  ├── 武侠世界（后续规划）
  ├── 克苏鲁世界（后续规划）
  └── ...（持续扩展）
```

> 每个世界是一个独立智能体：独立的世界观设定、独立的 System Prompt、独立的玩法规则。平台负责通用能力（用户、存档、消息持久化），世界负责叙事与规则判定。

### 1.3 MVP 首个世界：修仙（五域）

- **智能体名称**：天道
- **世界设定**：五域修仙界（详见 design-inspiration.md）
- **核心 slogan**：千人千命，天道为笔。

### 1.4 技术决策速览

| 维度 | 决策 |
|---|---|
| 产品形态 | 纯网页（HTML + CSS + Vanilla JS），无构建步骤 |
| 后端 | Python FastAPI |
| 数据库 | SQLite（单文件零配置） |
| AI 模型 | DeepSeek v4 Pro/Flash、通义千问系列（国产优先） |
| 用户系统 | 注册/登录，尽可能简单；**主页免登录可预览** |
| 持久性 | 长期游玩，跨 session 保存进度 |
| MVP 范围 | 主页（预览）+ 注册/登录 + 修仙世界全体验，不含排行榜与死亡判定 |

## 二、用户角色

| 角色 | 描述 |
|---|---|
| **访客** | 未登录用户，可浏览主页、预览世界介绍与示例叙事，不可游玩 |

## 三、功能需求（按模块拆分）

### 3.0 主页模块（免登录）

#### FR-HOME-001 平台主页
- **访问**：网站根路径 `/`，无需登录
- **内容**：
  - 平台名称 "无敌 (Invencia)" + 一句话简介
  - 可用世界列表（MVP 仅修仙世界，展示名称、封面图/视觉、简介）
  - 世界详情预览（点击展开）：
    - 世界观简介（五域地图示意、境界体系一览）
    - 示例叙事片段（预设 1-2 段静态文本，展示叙事风格）
    - 玩法特色（自由文本输入、千人千命、无数字沉浸）
  - **"开始修仙"按钮** → 未登录则弹起登录/注册入口
- **风格**：黑底古风，大气排版，类似游戏官网 Landing Page

#### FR-HOME-002 登录/注册入口
- **触发时机**：
  - 点击主页"开始修仙"按钮
  - 访问任何需要认证的页面（如 `/characters`、`/game/*`）
- **交互**：弹窗或独立页面，包含登录表单 + "没有账号？注册" 入口
- **登录后行为**：弹出角色列表弹窗（无角色则跳转创建页）

#### FR-HOME-003 免登录预览保护
- **规则**：访客仅可浏览主页内容，`/characters`、`/game/*` 等路径访问时强制跳转登录
- **前端实现**：路由守卫，检查 Token 是否存在且有效

### 3.1 认证模块

#### FR-AUTH-001 用户注册
- **输入**：用户名、密码（密码需确认）
- **规则**：
  - 用户名 2-20 字符，仅允许中文、英文、数字、下划线
  - 密码 6-32 字符
  - 用户名唯一
- **输出**：注册成功 → 自动登录并跳转角色列表页
- **异常**：用户名已存在、密码过短/过长

#### FR-AUTH-002 用户登录
- **输入**：用户名、密码
- **规则**：
  - 密码哈希验证（bcrypt）
  - 登录成功颁发 JWT Token（有效期 7 天）
- **输出**：登录成功 → 跳转角色列表页
- **异常**：用户名不存在、密码错误

#### FR-AUTH-003 Token 自动续期
- **规则**：请求携带有效 Token 时自动刷新过期时间
- **实现**：JWT 中间件在验证通过后，若距离过期不足 24 小时则签发新 Token

#### FR-AUTH-004 登录态持久化
- **规则**：前端存储 JWT Token（localStorage），关闭浏览器后保持登录态

### 3.2 角色模块

#### FR-CHAR-001 角色列表弹窗
- **触发**：登录后自动弹出；也可从游戏界面导航栏入口打开
- **展示**：已创建的角色列表（MVP 上限 3 个），每项显示角色名、性别、境界简述、最后游玩时间
- **操作**：点击角色 → 进入游戏；点击“创建新角色” → 弹出创建角色弹窗
- **空状态**：无角色时列表为空，仅显示“创建新角色”按钮和引导文案

#### FR-CHAR-002 创建角色
- **输入**：角色名（必填）、性别（男/女，必选）
- **规则**：
  - 每个用户最多创建 3 个角色，超过上限时禁用创建按钮
  - 角色名 1-12 字符，仅允许中文
  - 其余全部（出身、地点、执念、机缘、初始修为）由天道 AI 随机生成
  - AI 生成完整的 `infomation` JSON（10 个子字段，均为叙事描述）
  - 生成时需传入系统初始 Prompt（世界观 + infomation 结构 + 初始修为约束：炼气期初窥门径）
- **输出**：生成的角色面板（infomation 全文展示）
- **异常**：角色名重复（同一用户下）、AI 生成失败重试

#### FR-CHAR-003 查看角色面板
- **内容**：以古风玉简/卷轴风格展示 infomation 全部 10 个子字段
- **字段**：realm（境界）、power（战力）、origin（出身）、obsession（执念）、location（位置）、dao_heart（道心）、causality（因果）、relationships（人际关系）、inventory（物品）、summary（一句话概要）

#### FR-CHAR-004 角色面板实时更新
- **规则**：每次 `/game/act` 返回后，前端获取最新的 infomation 并在面板中刷新


#### FR-CHAR-005 按世界查询角色列表
- **描述**：根据世界 ID 查询当前用户在该世界下的所有角色
- **输入**：`world_id`（可选，不传则返回全部世界的角色）
- **规则**：与 FR-CHAR-001 共用同一接口 `GET /api/characters`，通过 query 参数区分
- **输出**：角色简要列表（id、name、gender、realm_summary、updated_at）
- **使用场景**：角色列表弹窗按世界分类展示

### 3.3 游戏核心循环模块（最重要）

#### FR-GAME-001 自由文本输入
- **描述**：玩家在输入框中通过自然语言表达决策意图
- **规则**：
  - 文本长度 1-2000 字符
  - 支持中文及标点
  - 输入不能为空或纯空白
- **示例**：
  - "我悄悄绕到妖兽背后，试图用刚学会的破风剑诀偷袭它的左翼。"
  - "先不急着进洞，我蹲在洞口观察半个时辰，看看有没有东西进出。"

#### FR-GAME-002 天道 AI 叙事生成
- **描述**：将玩家输入 + 上下文发送给 AI，由 AI 生成叙事、推动剧情
- **上下文组装规则**：
  1. System Prompt：世界观 + 角色 infomation 全文 + 叙事规则 + 输出格式要求
  2. 历史消息：取最近 N 条对话（N=20，可配置）
  3. 当前玩家输入
- **AI 输出要求**：
  - 一段沉浸式叙事（500-2000 字符）
  - 一个开放式钩子（暗示 2-4 种可能方向）
  - infomation 增量更新（仅返回有变化的字段）
- **异常处理**：AI 调用超时/失败 → 重试 1 次 → 仍失败返回友好错误提示

#### FR-GAME-003 战斗判定
- **描述**：AI 根据角色境界 + 文气强弱 + 策略合理性 + 环境因素进行战斗叙事
- **规则**：
  - 同境界内胜负由文气强弱与策略质量决定
  - 跨境界战斗境界压制优先，但保留绝境翻盘的叙事空间
  - 战斗结果由 AI 判定并融入叙事，不返回数值

#### FR-GAME-004 渡劫判定
- **MVP 规则**：渡劫期仅叙事，不执行永久死亡
- **后续迭代**：引入高风险死亡判定

#### FR-GAME-005 infomation 增量维护
- **描述**：每次 AI 响应中解析 infomation 变更，增量更新数据库
- **规则**：
  - AI 返回变更字段列表
  - 后端 merge 进 t_character.infomation
  - 确保 10 个子字段始终完整（不允许字段缺失）
- **示例**：
  - 玩家获得一把剑 → inventory 字段追加描述
  - 玩家突破境界 → realm、power、summary 同时更新

#### FR-GAME-006 对话历史展示
- **描述**：游戏界面展示历史对话，类似聊天界面
- **展示格式**：
  - 天道叙事：古风排版，白色/金色文字
  - 玩家输入：右对齐，灰色文字
  - 系统事件（突破、遭遇等）：居中，特殊样式
- **滚动加载**：每次进入游戏加载最近 50 条消息，向上滚动加载更早消息

#### FR-GAME-007 对话历史持久化
- **规则**：
  - 每条玩家输入和 AI 叙事均写入 t_message
  - message.type = "user" / "assistant" / "system"
  - 关闭浏览器后回来可继续（自动加载上次对话位置）


#### FR-AUX-001 全局错误提示
- 网络错误、AI 超时、服务端异常均以 Toast 形式展示友好提示
- 不可恢复的错误（如 AI 持续失败）提供"重试"按钮

#### FR-AUX-002 Loading 状态
- AI 生成叙事期间（预计 3-15 秒），展示 Loading 动画（古风风格：如"天道推演中..."）
- 输入框在 AI 响应返回前禁用

## 四、非功能性需求

### 4.1 性能

| 指标 | 目标 |
|---|---|
| AI 响应时间 | 普通叙事 3-8 秒，复杂判定 8-15 秒 |
| 页面首屏加载 | < 2 秒（纯静态 HTML，无构建） |
| 历史消息加载 | 50 条 < 500ms |
| 数据库查询 | 单次查询 < 100ms |

### 4.2 安全

| 要求 | 说明 |
|---|---|
| 密码存储 | bcrypt 哈希，不可逆 |
| 认证 | JWT Token，7 天过期 |
| 输入校验 | 前端 + 后端双重校验，防 XSS（输入纯文本，不渲染 HTML） |
| AI 输出安全 | AI 返回内容仅作文本展示，不解析 HTML/JS |
| SQL 注入防护 | 使用 ORM 参数化查询 |
| 路由保护 | 前端路由守卫 + 后端中间件双重校验，未登录不可访问游戏接口 |

### 4.3 体验

| 要求 | 说明 |
|---|---|
| UI 品质 | 上线级——黑底古风、文字排版考究、交互动画流畅 |
| 移动端响应式 | 支持手机竖屏阅读（主要使用场景） |
| 文字可读性 | 正文字号 ≥ 16px，行高 ≥ 1.8 |
| 空状态 | 新用户角色列表为空时展示引导文案 |
| 错误恢复 | AI 失败后允许玩家重试（重新发送相同输入） |

### 4.4 兼容性

| 浏览器 | 版本 |
|---|---|
| Chrome | ≥ 90 |
| Edge | ≥ 90 |
| Firefox | ≥ 90 |
| Safari | ≥ 14 |
| 移动端浏览器 | iOS Safari ≥ 14, Android Chrome ≥ 90 |

### 4.5 可维护性

| 要求 | 说明 |
|---|---|
| 前端零构建 | 纯 HTML/CSS/JS，直接打开/部署 |
| 后端模块化 | auth / game / character / world 独立模块 |
| 配置外部化 | AI API Key、模型名称等敏感/可变配置通过环境变量或 config.py 管理 |
| 世界可扩展 | 新增世界只需添加 System Prompt 配置 + 世界介绍页内容，无需改动核心引擎 |

## 五、核心用户流程

### 5.1 访客预览流程

```
访问网站 `/`
  → 平台主页（展示修仙世界预览：世界观、境界、示例叙事）
    → 点击"开始修仙"
      → 未登录 → 登录/注册弹窗
        → 注册/登录成功
          → 角色列表页（空状态）
            → 创建角色 ...（进入 5.2 流程）
```

### 5.2 首次游玩流程

```
创建角色页（输入角色名 + 选择性别）
  → 提交 → AI 生成出身 + infomation（Loading："天道正在编织你的命运..."）
    → 角色面板展示（卷轴风格，全 infomation 预览）
      → 点击"踏入仙途"
        → 游戏主界面（第一段天道开场叙事 + 输入框）
          → 输入决策 → AI 驱动叙事 → 持续循环
```

### 5.3 回归游玩流程

```
访问网站 `/` → 主页（已登录状态，导航栏显示"我的角色"）
  → 角色列表页
    → 点击角色 → 游戏主界面
      → 自动加载最近对话历史 + 最新 infomation
        → 输入决策 → 继续游玩
```

### 5.4 异常流程

- **AI 生成失败**：显示错误提示 + "重试"按钮，不允许继续游戏
- **网络断开**：前端检测 → 提示"天道暂时无法回应"，输入框禁用
- **Token 过期**：自动跳转登录页
- **访客访问受限页面**：前端路由守卫拦截 → 跳转主页并弹起登录

## 六、页面结构

精简为 3 个视图 + 3 个弹窗，减少页面维护成本。

### 视图

| 视图 | 路由 | 认证 | 说明 |
|---|---|---|---|
| **主页** | `/` | 无 | 平台 Landing Page：世界展示 + 示例叙事 |
| **登录/注册** | `/login` | 无 | 登录表单 + 注册入口；也可在主页弹窗实现 |
| **游戏主界面** | `/game/:character_id` | JWT | 核心视图，内含对话区 + 输入框 + 角色面板侧栏 |

### 弹窗

| 弹窗 | 触发条件 | 说明 |
|---|---|---|
| **登录/注册** | 主页点击“开始修仙”且未登录 | 登录表单 / 注册表单切换 |
| **角色列表** | 登录后进入游戏前 | 已有角色列表 + “创建新角色”按钮 |
| **创建角色** | 角色列表中点击“创建新角色” | 角色名 + 性别 + AI 生成 → 预览 → 踏入仙途 |

> 角色创建不需要独立页面（一次性引导流程，弹窗完成，MVP 上限 3 个）。角色详情不需要独立页面（侧边栏随时查看）。

## 七、开发阶段划分

### Phase 1：骨架搭建（预计 2-3 天）

- [ ] FastAPI 项目初始化（目录结构、配置、数据库连接）
- [ ] SQLite 表创建（四表：t_world / t_user / t_character / t_message）
- [ ] 前端页面 HTML 骨架 + CSS 基础样式（黑底古风基调）
- [ ] 前端路由切换逻辑（含免登录路由守卫）

### Phase 2：主页 + 认证模块（预计 2-3 天）

- [ ] 平台主页 UI（Landing Page 风格：世界展示 + 示例叙事预览）
- [ ] `GET /api/worlds` API（返回修仙世界配置）
- [ ] 注册/登录 API + JWT 中间件
- [ ] 登录/注册 UI（主页弹窗或独立页面）

## 八、风险分析

| 风险 | 影响 | 概率 | 缓解措施 |
|---|---|---|---|
| AI 响应时间过长 | 体验差 | 高 | 使用 Flash 模型（DeepSeek Flash）作为日常叙事主力；展示 Loading 动画缓解等待焦虑 |
| AI 输出格式不稳定 | infomation 更新失败 | 中 | 强约束输出格式；JSON 解析失败时降级处理（不更新 infomation，仅展示叙事） |
| AI 偏离世界观 | 叙事不一致 | 中 | System Prompt 中强化世界观约束；定期检查 infomation 一致性 |
| Token 消耗过大 | 成本高 | 低 | 控制历史消息轮数；使用 Flash 模型降低成本 |
| DeepSeek/千问 API 不可用 | 服务中断 | 低 | 至少配置两个模型提供商，支持自动 fallback |
| 多世界扩展复杂度 | 架构耦合 | 低 | 初期就按世界级 Prompt 分离设计，t_world 表预留扩展性 |

## 九、术语表

| 术语 | 定义 |
|---|---|
| **Invencia / 无敌** | 平台名称，承载多个文字 RPG 世界 |
| **世界** | 一个独立的叙事 RPG 智能体，有独立世界观和规则 |
| **天道** | 修仙世界的 AI 智能体，驱动全部叙事与判定 |
| **infomation** | 角色信息载体，10 个子字段全叙事描述，零数值 |
| **文气强弱** | 同境界内的微妙战力区分（初窥门径/渐入佳境/炉火纯青/登峰造极），纯叙事 |
| **钩子** | 每段叙事结尾的开放式提示，暗示 2-4 种可能方向 |
| **五域** | 修仙世界的五块大陆设定 |
| **info_changes** | AI 响应中的 infomation 增量更新数据 |
