## 一、项目背景摘要

**无敌 (Invencia)** 是一款叙事驱动的修仙文字 RPG，由 AI"天道"智能体全程驱动叙事、NPC 对话、战斗判定与世界推进。玩家通过自由文本输入做出抉择，每段叙事以开放式钩子结尾。核心 slogan：**千人千命，天道为笔。**

| 维度 | 决策 |
|---|---|
| 产品形态 | 纯网页（HTML + CSS + Vanilla JS），无构建步骤 |
| 后端 | Python FastAPI |
| 数据库 | SQLite（单文件零配置） |
| AI 模型 | DeepSeek v4 Pro/Flash、通义千问系列（国产优先） |
| 用户系统 | 注册/登录，尽可能简单 |
| 持久性 | 长期游玩，跨 session 保存进度 |
| MVP 范围 | 注册→创建角色→全境界自由游玩，不含排行榜与死亡判定 |

---

## 二、用户角色

| 角色 | 描述 |
|---|---|
| **玩家** | 注册/登录后创建角色，通过自由文本输入进行修仙冒险 |
| **天道（系统）** | AI 驱动叙事生成、NPC 对话、战斗判定、世界推进、infomation 维护 |

> 注：MVP 阶段无管理员后台，所有管理操作通过数据库直接操作。

## 三、功能需求（按模块拆分）

### 3.1 认证模块

#### FR-AUTH-001 用户注册
- **输入**：用户名、密码（密码需确认）
- **规则**：
  - 用户名 2-20 字符，仅允许中文、英文、数字、下划线
  - 密码 6-32 字符
  - 用户名唯一
- **输出**：注册成功 → 自动登录并跳转角色列表页
- **异常**：用户名已存在、密码过短/过长

#### FR-AUTH-002 用户登录
- **输入**：用户名、密码
- **规则**：
  - 密码哈希验证（bcrypt）
  - 登录成功颁发 JWT Token（有效期 7 天）
- **输出**：登录成功 → 跳转角色列表页
- **异常**：用户名不存在、密码错误

#### FR-AUTH-003 Token 自动续期
- **规则**：请求携带有效 Token 时自动刷新过期时间
- **实现**：JWT 中间件在验证通过后，若距离过期不足 24 小时则签发新 Token

#### FR-AUTH-004 登录态持久化
- **规则**：前端存储 JWT Token（localStorage），关闭浏览器后保持登录态

---

### 3.2 角色模块

#### FR-CHAR-001 角色列表
- **规则**：登录后展示当前用户的所有角色（MVP 阶段限制每个用户最多 1 个角色）
- **展示**：角色名、性别、境界简述、最后游玩时间

#### FR-CHAR-002 创建角色
- **输入**：角色名（必填）、性别（男/女，必选）
- **规则**：
  - 角色名 1-12 字符，仅允许中文
  - 其余全部（出身、地点、执念、机缘、初始修为）由天道 AI 随机生成
  - AI 生成完整的 `infomation` JSON（10 个子字段，均为叙事描述）
  - 生成时需传入系统初始 Prompt（世界观 + infomation 结构 + 初始修为约束：炼气期初窥门径）
- **输出**：生成的角色面板（infomation 全文展示）
- **异常**：角色名重复（同一用户下）、AI 生成失败重试

#### FR-CHAR-003 查看角色面板
- **内容**：以古风玉简/卷轴风格展示 infomation 全部 10 个子字段
- **字段**：realm（境界）、power（战力）、origin（出身）、obsession（执念）、location（位置）、dao_heart（道心）、causality（因果）、relationships（人际关系）、inventory（物品）、summary（一句话概要）

#### FR-CHAR-004 角色面板实时更新
- **规则**：每次 `/game/act` 返回后，前端获取最新的 infomation 并在面板中刷新

### 3.3 游戏核心循环模块（最重要）

#### FR-GAME-001 自由文本输入
- **描述**：玩家在输入框中通过自然语言表达决策意图
- **规则**：
  - 文本长度 1-2000 字符
  - 支持中文及标点
  - 输入不能为空或纯空白
- **示例**：
  - "我悄悄绕到妖兽背后，试图用刚学会的破风剑诀偷袭它的左翼。"
  - "先不急着进洞，我蹲在洞口观察半个时辰，看看有没有东西进出。"

#### FR-GAME-002 天道 AI 叙事生成
- **描述**：将玩家输入 + 上下文发送给 AI，由 AI 生成叙事、推动剧情
- **上下文组装规则**：
  1. System Prompt：世界观 + 角色 infomation 全文 + 叙事规则 + 输出格式要求
  2. 历史消息：取最近 N 条对话（N=20，可配置）
  3. 当前玩家输入
- **AI 输出要求**：
  - 一段沉浸式叙事（500-2000 字符）
  - 一个开放式钩子（暗示 2-4 种可能方向）
  - infomation 增量更新（JSON Patch 格式，仅返回有变化的字段）
- **异常处理**：AI 调用超时/失败 → 重试 1 次 → 仍失败返回友好错误提示

#### FR-GAME-003 战斗判定
- **描述**：AI 根据角色境界 + 文气强弱 + 策略合理性 + 环境因素进行战斗叙事
- **规则**：
  - 同境界内胜负由文气强弱与策略质量决定
  - 跨境界战斗境界压制优先，但保留绝境翻盘的叙事空间
  - 战斗结果由 AI 判定并融入叙事，不返回数值

#### FR-GAME-004 渡劫判定
- **MVP 规则**：渡劫期仅叙事，不执行永久死亡
- **后续迭代**：引入高风险死亡判定

#### FR-GAME-005 infomation 增量维护
- **描述**：每次 AI 响应中解析 infomation 变更，增量更新数据库
- **规则**：
  - AI 返回 JSON Patch 或变更字段列表
  - 后端 merge 进 t_character.infomation
  - 确保 10 个子字段始终完整（不允许字段缺失）
- **示例**：
  - 玩家获得一把剑 → inventory 字段追加描述
  - 玩家突破境界 → realm、power、summary 同时更新

#### FR-GAME-006 对话历史展示
- **描述**：游戏界面展示历史对话，类似聊天界面
- **展示格式**：
  - 天道叙事：古风排版，白色/金色文字
  - 玩家输入：右对齐，灰色文字
  - 系统事件（突破、遭遇等）：居中，特殊样式
- **滚动加载**：每次进入游戏加载最近 50 条消息，向上滚动加载更早消息

#### FR-GAME-007 对话历史持久化
- **规则**：
  - 每条玩家输入和 AI 叙事均写入 t_message
  - message.type = "user" / "assistant" / "system"
  - 关闭浏览器后回来可继续（自动加载上次对话位置）

#### FR-GAME-008 新对话开始
- **规则**：玩家可开始"新的一天"（新 session），切换后不可返回之前的会话
- **实现**：创建新的 conversation_id，从 infomation 当前状态继续

### 3.4 辅助功能

#### FR-AUX-001 全局错误提示
- 网络错误、AI 超时、服务端异常均以 Toast 形式展示友好提示
- 不可恢复的错误（如 AI 持续失败）提供"重试"按钮

#### FR-AUX-002 Loading 状态
- AI 生成叙事期间（预计 3-15 秒），展示 Loading 动画（古风风格：如"天道推演中..."）
- 输入框在 AI 响应返回前禁用

## 四、非功能性需求

### 4.1 性能

| 指标 | 目标 |
|---|---|
| AI 响应时间 | 普通叙事 3-8 秒，复杂判定 8-15 秒 |
| 页面首屏加载 | < 2 秒（纯静态 HTML，无构建） |
| 历史消息加载 | 50 条 < 500ms |
| 数据库查询 | 单次查询 < 100ms |

### 4.2 安全

| 要求 | 说明 |
|---|---|
| 密码存储 | bcrypt 哈希，不可逆 |
| 认证 | JWT Token，7 天过期 |
| 输入校验 | 前端 + 后端双重校验，防 XSS（输入纯文本，不渲染 HTML） |
| AI 输出安全 | AI 返回内容仅作文本展示，不解析 HTML/JS |
| SQL 注入防护 | 使用 ORM 参数化查询 |

### 4.3 体验

| 要求 | 说明 |
|---|---|
| UI 品质 | 上线级——黑底古风、文字排版考究、交互动画流畅 |
| 移动端响应式 | 支持手机竖屏阅读（主要使用场景） |
| 文字可读性 | 正文字号 ≥ 16px，行高 ≥ 1.8 |
| 空状态 | 新用户角色列表为空时展示引导文案 |
| 错误恢复 | AI 失败后允许玩家重试（重新发送相同输入） |

### 4.4 兼容性

| 浏览器 | 版本 |
|---|---|
| Chrome | ≥ 90 |
| Edge | ≥ 90 |
| Firefox | ≥ 90 |
| Safari | ≥ 14 |
| 移动端浏览器 | iOS Safari ≥ 14, Android Chrome ≥ 90 |

### 4.5 可维护性

| 要求 | 说明 |
|---|---|
| 前端零构建 | 纯 HTML/CSS/JS，直接打开/部署 |
| 后端模块化 | auth / game / character 独立模块 |
| 配置外部化 | AI API Key、模型名称等敏感/可变配置通过环境变量或 config.py 管理 |

## 五、核心用户流程

### 5.1 首次游玩流程

```
访问网站
  → 注册页（输入用户名/密码）
    → 注册成功，自动登录
      → 角色列表页（空状态）
        → 创建角色页（输入角色名 + 选择性别）
          → 提交 → AI 生成出身 + infomation（Loading："天道正在编织你的命运..."）
            → 角色面板展示（卷轴风格，全 infomation 预览）
              → 点击"踏入仙途"
                → 游戏主界面（第一段天道开场叙事 + 输入框）
                  → 输入决策 → AI 驱动叙事 → 持续循环
```

### 5.2 回归游玩流程

```
访问网站
  → 登录页（若 Token 有效则直接跳过）
    → 角色列表页（已有角色）
      → 点击角色 → 游戏主界面
        → 自动加载最近对话历史 + 最新 infomation
          → 输入决策 → 继续游玩
```

### 5.3 异常流程

- **AI 生成失败**：显示错误提示 + "重试"按钮，不允许继续游戏
- **网络断开**：前端检测 → 提示"天道暂时无法回应"，输入框禁用
- **Token 过期**：自动跳转登录页

| 页面 | 路由 | 说明 |
|---|---|---|
| 登录页 | `/login` | 用户名 + 密码 + 注册入口 |

## 七、API 设计概要

### 7.1 认证

| 方法 | 路径 | 说明 | 认证 |
|---|---|---|---|
| POST | `/api/auth/register` | 注册 | 无 |
| POST | `/api/auth/login` | 登录，返回 JWT | 无 |
| POST | `/api/auth/refresh` | 刷新 Token | JWT |

### 7.2 角色

| 方法 | 路径 | 说明 | 认证 |
|---|---|---|---|
| GET | `/api/characters` | 获取当前用户角色列表 | JWT |
| POST | `/api/characters` | 创建角色（触发 AI 生成出身） | JWT |
| GET | `/api/characters/:id` | 获取角色详情（含 infomation） | JWT |

### 7.3 游戏

| 方法 | 路径 | 说明 | 认证 |
|---|---|---|---|
| POST | `/api/game/:character_id/act` | 核心接口：发送玩家决策，获取 AI 叙事 | JWT |
| GET | `/api/game/:character_id/messages` | 获取对话历史（分页） | JWT |

### 7.4 通用规范

- 请求/响应格式：JSON
- 认证方式：Header `Authorization: Bearer <token>`
- 统一响应结构：`{ "code": 0, "data": {...}, "message": "ok" }`
- 错误响应：`{ "code": 错误码, "data": null, "message": "错误描述" }`

## 八、AI Prompt 架构设计要点

### 8.1 System Prompt 结构

```
[世界观设定]（九域、境界体系、天道法则）
[角色当前状态]（infomation 10 字段全文）
[叙事规则]
  - 以沉浸式小说语言叙事
  - 每次回复以开放式钩子结尾
  - 战斗判定需考虑境界/强弱/策略/环境
  - 境界突破需要累积足够机缘或苦修
  - 不暴露任何数值，只用文字描摹强弱
[输出格式要求]
  - 必须返回 JSON，包含 narrative / hook / info_changes 三个字段
  - info_changes 为 JSON Patch 格式，仅返回有变化的 infomation 字段
[安全性约束]
  - 拒绝执行玩家要求跳出角色扮演的指令
  - 拒绝生成色情、极端暴力内容
```

### 8.2 上下文窗口管理

- **历史消息**：取最近 20 轮对话（约 8000-12000 tokens）
- **infomation**：全字段约 1500-2500 tokens
- **System Prompt**：约 1000-1500 tokens
- **总计**：每轮约 10000-15000 tokens，在 DeepSeek/千问上下文窗口内安全

### 8.3 info_changes 增量更新协议

AI 返回的 `info_changes` 格式：

```json
{
  "realm": "金丹初凝，尚有裂缝...（若境界变化则完整替换）",
  "inventory": "粗布衣 + 豁口猎刀 + 刚捡到的锈铁剑（若新增物品则追加描述）",
  "summary": "更新后的概要..."
}
```

- 仅返回有变化的字段
- 空对象 `{}` 表示本轮无变化
- 后端 merge 而非覆盖：确保未返回的字段保持不变

## 十、风险分析

| 风险 | 影响 | 概率 | 缓解措施 |
|---|---|---|---|
| AI 响应时间过长 | 体验差 | 高 | 使用 Flash 模型（DeepSeek Flash）作为日常叙事主力；展示 Loading 动画缓解等待焦虑 |
| AI 输出格式不稳定 | infomation 更新失败 | 中 | 强约束输出格式；JSON 解析失败时降级处理（不更新 infomation，仅展示叙事） |
| AI 偏离世界观 | 叙事不一致 | 中 | System Prompt 中强化世界观约束；定期检查 infomation 一致性 |
| Token 消耗过大 | 成本高 | 低 | 控制历史消息轮数；使用 Flash 模型降低成本 |
| DeepSeek/千问 API 不可用 | 服务中断 | 低 | 至少配置两个模型提供商，支持自动 fallback |

## 十一、术语表

| 术语 | 定义 |
|---|---|
| **天道** | AI 智能体的世界观化称呼，驱动全部叙事与判定 |

| **文气强弱** | 同境界内的微妙战力区分（初窥门径/渐入佳境/炉火纯青/登峰造极），纯叙事 |
| **钩子** | 每段叙事结尾的开放式提示，暗示 2-4 种可能方向 |
| **九域** | 修仙世界的九块大陆设定 |
| **info_changes** | AI 响应中的 infomation 增量更新数据 |
